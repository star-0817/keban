# 课伴 AI 协作手册

本手册规定“课伴”项目中由多个 AI 协作开发时的任务拆分、分支、交付和验收方式。目标是让任务可独立完成、可检查、可安全合并。

## 1. 角色分工

| 角色 | 责任 | 是否直接写业务代码 |
| --- | --- | --- |
| 项目协调 AI | 拆分 Issue、明确边界、审查 PR、处理冲突、安排集成和发布 | 视集成需要而定 |
| 模块开发 AI | 在单独分支内完成一个 Issue 的实现和测试 | 是 |
| 测试 AI | 根据验收标准做自动化/手工验证，提交缺陷 Issue | 原则上否 |
| 设计 AI | 提供页面结构、交互状态、文案和视觉规范 | 原则上否 |

当前由项目协调 AI 维护架构、分工和最终验收。用户负责确认产品方向、提供真机测试反馈，以及决定涉及账号、支付、广告或对外发布的事项。

## 2. 不可违反的工程约束

- 第一版目标仅为 Android App。
- 使用 UniApp、Vue 3、TypeScript 和 Pinia；业务数据通过 SQLite 存储。
- 核心能力必须离线可用，不依赖自建服务器或账号体系。
- 不上传用户学生名单、成绩、文件、图片、录音等数据。
- 新增业务逻辑必须有对应单元测试；需要原生设备能力的功能必须给出真机测试清单。
- 页面不得直接读写 SQLite，也不得直接调用 Android API；必须分别经由仓储层和类型化插件包装层。
- 不把图片、PDF、录音等二进制内容存入 SQLite，只保存路径或 Content URI 与元数据。
- 未经任务明确授权，不引入广告、统计 SDK、云端 API、账号或支付能力。

## 3. GitHub 工作流

### 3.1 Issue 模板

每个开发任务创建一个 Issue，内容必须包含：

```text
标题：<模块>：<可交付能力>

背景：用户在什么场景下遇到什么问题。
目标：这次要交付什么。
范围：允许修改的目录/文件、需要新增的内容。
不做什么：明确排除项。
验收标准：可观察、可测试的完成条件。
测试要求：单元测试、组件测试或真机手工步骤。
依赖：必须先完成的 Issue；没有则写“无”。
```

### 3.2 分支和提交

从最新 `main` 创建分支：

```text
feat/12-roster-domain
fix/23-pdf-output-name
docs/4-architecture-rules
```

提交信息使用 Conventional Commits：

```text
feat(class): add roster validation
test(study): cover GPA conversion rules
fix(document): preserve source PDF files
docs: clarify Android-only scope
```

### 3.3 Pull Request 要求

PR 描述必须填报：

```text
关联 Issue：Closes #<编号>
实现内容：
- ...

检查：
- 命令：...
- 结果：通过/失败（说明原因）

手工验证：
- 设备/模拟器：...
- 步骤与结果：...

风险或限制：
- ...
```

合并前要求：

- 分支基于最新 `main`，不存在未解决冲突。
- 通过与本次修改相关的测试、类型检查和代码格式检查。
- 不含密钥、个人隐私数据、构建产物或未经授权的大型二进制文件。
- 项目协调 AI 审查通过。

## 4. 代码边界

```text
src/
  domain/            纯业务模型、规则、计算器和校验器
  repositories/      SQLite 数据读写；不得包含 UI 逻辑
  services/          跨实体用例和事务编排
  stores/            Pinia 状态及对服务层的调用
  pages/             页面组合；不得直接访问数据库或原生 API
  components/        可复用展示和交互组件
  plugins/           类型化的 Android/UniApp 平台能力封装
  types/             跨模块共享类型
  utils/             无状态通用工具函数
tests/               与 src 结构大致对应的自动化测试
docs/                架构、产品、测试和协作说明
```

依赖方向只能从外向内：`pages/components -> stores -> services -> repositories/domain/plugins`。`domain` 不依赖 UniApp、Pinia、SQLite 或 Android API。

## 5. 里程碑与并行边界

| 里程碑 | 目标 | 可并行的工作 | 合并门槛 |
| --- | --- | --- | --- |
| M0 基础工程 | 建立可运行 Android 工程和基础规范 | 项目脚手架、设计令牌、领域类型、数据库方案调研 | 可安装调试包，示例数据重启后仍存在 |
| M1 班级工具 | 名单、抽人、分组与历史 | 名单领域逻辑、班级页面、CSV 解析、抽人算法测试 | 20/60 人名单通过验收 |

| M3 文档工具 | 封面、图片转 PDF、PDF 合并排序 | 封面表单/模板、文件选择包装、PDF 插件调研 | 真机输出文件可打开且源文件不变 |

## 7. AI 任务提示词通用模板

将下列内容复制给负责某个 Issue 的 AI，并替换方括号部分：

```text
你是“课伴”Android App 的模块开发 AI。请只完成 GitHub Issue #[编号]：[标题]。

项目背景：课伴是面向在校学生与班委的 Android 优先、离线优先校园工具 App。技术栈为 UniApp、Vue 3、TypeScript、Pinia、SQLite；不使用自建服务器，不强制登录，不上传用户数据。

开始前：
1. 阅读 README.md、docs/DEVELOPMENT.md、docs/AI_COLLABORATION.md。
2. 检查当前分支和 git 状态；从最新 main 创建或切换到分支 `[分支名]`。
3. 只修改本 Issue 所需文件；若发现需要扩大范围，停止实现并说明原因。

Issue 内容：
- 背景：[背景]
- 目标：[目标]
- 范围：[允许修改的目录/文件]
- 不做什么：[排除项]
- 验收标准：[逐条列出]
- 依赖：[Issue 编号或“无”]

工程要求：
- 使用 TypeScript；业务逻辑保持可测试且不直接依赖页面或 Android API。
- 先添加/更新测试，再写实现；无法自动测试的原生能力需写出真机验证步骤。
- 不修改无关文件，不添加密钥、广告、云端 API 或用户跟踪。
- 运行与本任务相关的测试、类型检查和格式检查。

完成后：
1. 提交清晰的 Git 提交，格式使用 Conventional Commits。
2. 不要直接合并 main；创建 Pull Request 或提供可创建 PR 的完整信息。
3. 最终回复必须包含：修改文件、实现说明、测试命令与结果、真机验证（如适用）、已知限制、建议下一步。
```

## 8. 协调 AI 的审查清单

- Issue 目标与实际改动一致，边界没有扩张。
- 类型、名称、时间格式和错误处理符合项目约定。
- 新的业务规则有覆盖正常路径、边界条件和失败路径的测试。
- UI 明确说明本地处理和权限用途，不阻塞离线核心流程。
- 文档/PDF 操作不覆盖源文件；文件 URI 不被直接存为二进制数据。
- 数据库变更包含可升级迁移和恢复考虑。
- PR 可独立回滚，不包含无关格式化或大范围重构。
